## Problem Statement

Chủ shop K-beauty quản lý 10-50 đơn order Hàn / tháng bằng Excel + Zalo. Pain quan trọng nhất:
- Không track được ai cọc / ai chưa, đơn nào đã đặt với Hàn, đơn nào về VN rồi
- Khách hỏi "đơn em đến đâu rồi" phải lục Zalo 5-10 phút
- Tỷ giá KRW/VND đổi → đôi lúc báo giá sai, lỗ vài trăm nghìn / đơn
- Phí ship trong Hàn (Olive Young → kho gom) + ship quốc tế (Hàn → VN) là 2 cost lines riêng, dễ tính sai profit

## Demand Evidence

Chủ shop = chính user của tool. Workflow 9 orders gần nhất tracked trong `order.xlsx` confirm pattern:
- Brands chính: Clio, Bioderma, Mediheal, Make p:rem, Torriden, Numbuzin, Dalba
- Suppliers chính: Olive Young, Lotte Duty Free, brand direct shops (clubclio.co.kr)
- FX rate consistent 18 VND/KRW trong period này
- Tỉ lệ có phone/address: ~10% (chỉ 1/9 orders có địa chỉ giao)
- Status thực tế: `구매` (đã mua), `문제` (vấn đề), rỗng (pending)
- Quantity: cả integer (1, 2) lẫn fractional (0.5)

**Demand chưa verify từ user khác** — chưa interview chủ shop K-beauty khác.

## Target User & Wedge

**MVP user:** chủ shop K-beauty (chính chủ design doc).
**Wedge:**
1. Multi-stage payment tracking (cọc → ship Hàn → tất toán)
2. KRW/VND FX lock tại order creation
3. Phân tách rõ Korean domestic shipping (한국 내 배송비) vs Korea-VN international shipping (한베 배송비)
4. Brand + supplier organized view (vì user thinks in brands)
5. Public UUID link cho khách xem status không cần login

## Constraints

- Solo developer, làm side-project song song chạy shop
- Budget hosting: Supabase free tier + Vercel hobby
- Stack đã lock bởi CLAUDE.md (không tranh luận lại)
- Money = `Decimal` ở Python, `NUMERIC` ở Postgres — không bao giờ `float`
- Korean text + Vietnamese text — Unicode UTF-8 throughout

## Premises

1. **Build justified** cho 10-50 đơn/tháng vì có 5 wedges ở trên không có off-the-shelf nào fit hoàn hảo. *(Chưa validate vs Sapo/Kiotviet — xem Open Questions.)*
2. **Schema extensible từ ngày 1**, UI MVP-focused. Tốn thêm 1-2 tuần build nhưng tránh DB migration trong 12 tháng tới.
3. **Multi-tenant ready, single-tenant active.** `shop_id` FK trên data tables. Auth chỉ admin của shop 1 lúc MVP. Mở rộng = thêm shop record + cấp credentials.
4. **Khách không cần login.** Public UUID link đủ.
5. **Money = Decimal**, FX rate snapshot tại order creation + lưu cả lịch sử trong `fx_rates` table.
6. **Products & customers normalized** dù MVP có thể nhập free text. Lý do: 6 tháng sau cần brand catalog / customer history thì có sẵn.

## Approaches Considered

### Approach A: Minimal (v1) — denormalized, 4 tables
- Tables: customer, orders, order_items, payments
- Free text mọi thứ
- ~3 tuần build
- Pro: ship nhanh
- Con: brand/supplier analytics đầu năm sau cần migration đau

### Approach B: Extensible schema, MVP UI (CHOSEN — v2)
- 11 tables normalized (shops, brands, suppliers, products, variants, customers, addresses, orders, order_items, shipments, payments, fx_rates + audit_log)
- UI chỉ build core flows
- ~4-5 tuần build
- Pro: future-proof DB, không migration trong 12 tháng
- Con: thêm code complexity, schema có tables "không dùng ngay"

### Approach C: Full multi-tenant SaaS + customer login
- B + customer authentication + RLS policies + billing-ready
- ~8-10 tuần build
- Pro: bán cho shop khác từ ngày 1 OK
- Con: 2x build time cho hypothesis chưa validate

## Recommended Approach: B

**Lý do:** user requested "design thừa cho extensibility". B đáp ứng đúng — schema cho future, UI cho hiện tại. Migration risk thấp nhất, build time hợp lý.

## Schema

See [`SCHEMA-PATCH.sql`](SCHEMA-PATCH.sql) for the complete migration-ready SQL. 16 tables total, RLS policies, indexes, triggers — ready to drop into `supabase/migrations/`.

### Tables (16)

| Group | Tables |
|---|---|
| Tenancy | `shops`, `shop_members` (role enum: owner/admin/staff) |
| Catalog | `brands`, `suppliers`, `products`, `product_variants` |
| Customers | `customers` (soft delete), `customer_contacts`, `addresses` |
| Orders | `orders` (with `shipment_id`, soft delete), `order_items` |
| Shipments | `shipments` (group Hàn→VN runs) |
| FX | `fx_rates` (history with effective dates) |
| Payments | `payment_methods`, `payments` (with `idempotency_key`) |
| Audit | `audit_log` (generic, JSONB changes) |

### Security: RLS day 1

Every tenant table có RLS policy: `shop_id IN (SELECT user_shop_ids())` where `user_shop_ids()` is a SQL function returning shops user is member of. Backend dùng JWT của user (không phải service_role) cho user-facing ops. Service_role chỉ dùng cho audit_log inserts và admin tasks.

### Computed values — SERVICE LAYER (NOT DB)

All financial calculations live in `backend/app/services/order_calculations.py::compute_order_totals(order, items, payments, fx_rate) -> OrderTotals`:

- `total_vnd` = Σ(unit_sale_price_vnd × quantity) for all items
- `cost_vnd` = Σ(unit_cost_krw × quantity × fx_rate) + (korean_shipping_krw × fx_rate)
- `profit_vnd` = `total_vnd` − `cost_vnd` − `international_shipping_vnd`
- `total_paid_vnd` = Σ(deposit + balance) − Σ(refund)
- `amount_owed_vnd` = `total_vnd` + `international_shipping_vnd` − `total_paid_vnd`
- `shipment.international_shipping_per_order` = allocate `total_international_cost_vnd` proportionally by order total

**Decimal math throughout — never float.** Regression fixture: 9 orders from `order.xlsx`, profit must match column `이익` exactly (see [`TEST-PLAN.md`](TEST-PLAN.md)).

### Status state machine — APPLICATION LAYER

Status transitions enforced in `backend/app/services/order_status.py::validate_transition(from_status, to_status)`. Allowed transitions:

```
pending → ordered → in_transit → arrived → delivered → completed
                                                    ↓
                                              (any) → problem → ordered/cancelled
                                              (any) → cancelled
```

Invalid transition raises `ApiError(422, "invalid_status_transition", ...)`. Every transition writes `audit_log` row.

### Idempotency: payments

`POST /api/v1/orders/{id}/payments` requires `Idempotency-Key` header (UUID v4). Server INSERTs với `idempotency_key`; same key trong cùng shop → `ON CONFLICT DO NOTHING`, return existing record. TTL: keys older than 24h are accepted again (cleanup via cron).

## Frontend Screens (MVP)

**Admin (Supabase Auth login):**
1. Dashboard — count by status, tổng VND đang chờ thu, tổng KRW đã đặt với Hàn tháng này, top brands
2. Customer list / detail (lịch sử đơn + contacts)
3. Order list (filter: status, brand, customer, date)
4. Order detail (timeline status + payments + items + nút share public link)
5. New order flow (chọn customer → thêm items (free text product name + brand + URL + KRW cost + VND sale price) → input Korean shipping + international shipping → confirm)
6. FX rate setting screen (set tỷ giá hiện tại, xem history)
7. Payment recording (deposit / balance / refund)

**Public (no auth):**
8. `/o/{token}` — order status + payment summary + items. **KHÔNG show số điện thoại, address chi tiết, profit, cost.**

## Risk & Tradeoff Notes

**Bạn đã chọn extensibility over speed.** Quan trọng để biết:
- v1 (4 tables, denormalized): 3 tuần build, brand analytics đầu năm 2027 cần DB migration
- v2 (12 tables, normalized — chosen): 4-5 tuần build, extensible đến 12 tháng tới
- v3 (full SaaS): 8-10 tuần, premature

Mỗi tuần extra build có cost cơ hội — 1 tuần dùng để chạy shop thật, talk to customer, validate market = potentially higher ROI hơn 1 tuần code schema mở rộng cho hypothesis.

## What I noticed về cách bạn nghĩ

- Khi xem file Excel của bạn xong, mình thấy bạn track profit cực kỹ (column 이익) — bạn đã hiểu rõ unit economics. Đó là dấu hiệu tốt của shop owner.
- Bạn dùng Korean column names mặc dù khách là Vietnamese. Bạn nghĩ trong Korean khi source, trong Vietnamese khi sell. Tool phải tôn trọng cái dual-language ấy.
- Khi mình hỏi build vs buy, bạn chọn skip. Khi mình hỏi MVP scope, bạn chọn extensibility. Pattern: bạn tin vào instinct build, không bị evaluation paralysis. Đó là speed advantage. Trade-off: không có safety net của evidence.
- Trong eng review, bạn approve tất cả 4 recommendations (shop_members, RLS day 1, idempotency, service-layer computed values) mà không pushback. Có thể vì recommendations rõ ràng — nhưng cũng đáng để bạn tự hỏi: bạn approve vì hợp lý, hay vì đó là path-of-least-resistance? Khi build, nếu RLS test setup khó hơn dự kiến, bạn có sẵn sàng quay lại design này không?
