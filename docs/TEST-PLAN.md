## Affected Routes / Modules

### Backend (FastAPI)
- `POST /api/v1/orders` — create order with items + FX lock + Korean shipping
- `PATCH /api/v1/orders/{id}/status` — state machine transition + audit log
- `GET /api/v1/orders/{id}` — full detail with computed totals
- `GET /api/v1/orders` — filtered list (status, customer, brand, date)
- `POST /api/v1/orders/{id}/payments` — record payment with idempotency
- `GET /api/v1/public/orders/{token}` — public read-only (NO PII leak)
- `POST /api/v1/fx-rates` — set new FX rate (close current, open new)
- `GET /api/v1/dashboard` — aggregate stats

### Frontend (React)
- Admin login flow
- New order creation flow (multi-step form)
- Order list with filters
- Order detail with timeline + payment recording
- Public order page (no auth)

## Critical Paths (must work or feature is broken)

1. **Create order with 2+ items, FX locked at creation, Korean shipping > 0**
2. **Multi-stage payment lifecycle**: cọc 30% → ship Hàn → tất toán
3. **Public link**: customer mở URL UUID, thấy status + items + payment summary, KHÔNG thấy PII
4. **FX rate change**: set new rate doesn't affect existing orders' fx_rate_krw_to_vnd snapshot
5. **Profit calculation matches `order.xlsx`** — regression fixture from 9 historic orders

## Key Interactions to Verify

### Order creation flow
- Auto-fill FX rate from current `fx_rates` row
- Adding/removing line items recalculates total
- Korean shipping field accepts 0
- Submit button disabled while POSTing (frontend idempotency layer)

### Payment recording
- Type dropdown (deposit / balance / refund)
- Refund creates negative payment record OR negative-amount marker
- Idempotency: same key → returns existing record, not duplicate
- Updating payment recalculates `amount_owed`

### Public order page
- URL: `/o/{public_token}`
- Shows: order status, items (name + qty + variant notes), payment history (amounts + dates), total + amount_owed
- Does NOT show: customer phone, address details, profit, cost_krw, cost_vnd

### Status workflow
- Valid transitions: state machine enforces (e.g., `quoted` → only `deposit_paid` or `cancelled`)
- Invalid transition returns 422 with clear message
- Each change → row in `audit_log`

## Edge Cases

### Pricing & math
- KRW × FX rounding (use Decimal, never float)
- Quantity = 0.5 (split order)
- Quantity > 1 with mixed variants
- FX rate 18.123456 (more precision than display)
- Total with all 0 (free gift)
- Refund > deposit (over-refund) → reject or warn?
- Negative profit (loss) — should display correctly

### Korean text / Unicode
- Brand name 한글 (e.g., "메디힐")
- Product name mixed Korean + English
- Customer name Vietnamese with tones
- Notes column with emoji 🎁

### Auth & multi-tenancy
- User A (shop A) tries to GET /orders of shop B → 403 OR 404 (no info leak)
- User A INSERT into shop B → blocked by RLS
- Token rotation (Supabase Auth refresh)
- Expired JWT → 401, frontend redirects to login

### Idempotency
- Same `idempotency_key` from same user → return existing record
- Same key from different user → reject (cross-user replay)
- Stale key (>24h) → reject

### Status & lifecycle
- Cancel order with deposit paid → refund flow triggered
- Status jump backward (paid_in_full → quoted) → block via state machine
- Multiple status changes in 1 minute → audit log preserves order via timestamp
